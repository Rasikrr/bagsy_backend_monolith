# Приглашение сотрудников (Flow из stage ветки)

> Документация существующего флоу для рефакторинга. Код анализировался из ветки `stage`.

## Общая схема

Приглашение сотрудника — **двухшаговый процесс** с **link-based токеном** (не OTP-код):

```
Менеджер/Владелец → Создаёт приглашение → Отправляется ссылка (WhatsApp/SMS)
                                                    ↓
Сотрудник → Переходит по ссылке → Устанавливает пароль → Зарегистрирован
```

## Шаг 1: Менеджер создаёт приглашение

**Эндпоинт:** `POST /api/v1/auth/staff/register` (требует авторизацию Manager+)

**Тело запроса:**
```json
{
  "phone": "+77001234567",
  "name": "Иван",
  "surname": "Иванов",
  "role": "staff",
  "point_code": "point-1"
}
```

**Что происходит:**

1. Извлекается `Actor` из контекста (кто приглашает).
2. Проверяются права:
   - `Staff` — **не может** приглашать.
   - `Manager` — может создавать только `Staff`, только в **свою точку** (тот же `PointCode`).
   - `NetManager`/`SelfOwner` — могут создавать `Manager` и `Staff` в любую точку **своей сети**.
3. Проверка дубликата — ищем в Redis `auth:staff:confirm:{phone}`. Если уже есть → `409 Conflict`.
4. Данные регистрации сохраняются в Redis (`auth:staff:confirm:{phone}`, TTL ~24 часа).
5. Генерируется **invite-токен** — 10 символов `[a-z0-9]`.
6. Токен сохраняется в Redis (`auth_token:{token}`) с метаданными (phone, point_code, network_code, purpose: "register").
7. Отправляется уведомление: WhatsApp → SMS fallback.
   - Текст: `"Добро пожаловать в Bagsy! Завершите регистрацию по ссылке: {url}/{token}"`

## Промежуточный шаг: Валидация токена

**Эндпоинт:** `GET /api/v1/auth/verify-auth-token/{token}` (без авторизации)

Сотрудник переходит по ссылке. Фронтенд вызывает этот эндпоинт чтобы:
- Проверить что токен валиден и не истёк.
- Получить контекст (телефон, точка, сеть) для отображения на форме.

**Ответ:**
```json
{
  "phone": "+77001234567",
  "network_code": "network-1",
  "point_code": "point-1",
  "purpose": "register"
}
```

Токен **не расходуется** на этом шаге (read-only).

## Шаг 2: Сотрудник завершает регистрацию

**Эндпоинт:** `POST /api/v1/auth/staff/register/confirm` (без авторизации, токен = auth)

**Тело запроса:**
```json
{
  "token": "abc123xyz0",
  "password": "securePassword123"
}
```

> Имя и фамилия **не** передаются — они были указаны менеджером на шаге 1.

**Что происходит:**

1. Ищем invite-токен в Redis (`auth_token:{token}`). Не найден → `401`.
2. Ищем данные регистрации в Redis (`auth:staff:confirm:{phone}`).
3. Внутри **транзакции БД**:
   - **Если пользователь не существует** → `CreateStaff()` — создаётся новый пользователь со всеми полями.
   - **Если пользователь существует** (как клиент с `RoleUser`) → `PromoteToStaff()` — обновляется роль, пароль, точка. Нет дубликатов.
4. Invite-токен удаляется из Redis (one-time use).
5. Генерируются JWT-токены (access + refresh).

**Ответ:**
```json
{
  "access_token": "eyJhbGci...",
  "refresh_token": "..."
}
```

## Повторная отправка ссылки

**Эндпоинт:** `POST /api/v1/auth/staff/register/resend` (Manager+)

```json
{
  "phone": "+77001234567"
}
```

1. Берёт существующие данные регистрации из Redis.
2. Генерирует **новый** токен.
3. Обновляет TTL на данных в Redis.
4. Повторно отправляет ссылку (WhatsApp → SMS).

Rate limit: 3 запроса в минуту.

## Эндпоинты (сводка)

| Метод | Эндпоинт | Авторизация | Назначение |
|-------|----------|-------------|------------|
| `POST` | `/api/v1/auth/staff/register` | Manager+ | Создать приглашение |
| `GET` | `/api/v1/auth/verify-auth-token/{token}` | Нет | Валидация токена |
| `POST` | `/api/v1/auth/staff/register/confirm` | Нет (токен) | Завершить регистрацию |
| `POST` | `/api/v1/auth/staff/register/resend` | Manager+ | Повторить отправку |

## Верификация

**Не используется OTP.** Используется link-based одноразовый токен:
- 10 символов `[a-z0-9]`
- Хранится в Redis с TTL ~24 часа
- Расходуется при confirm

Это отличается от регистрации владельца, где используется 4-значный OTP-код.

## Уведомления

- **Первичный канал:** WhatsApp (GreenAPI)
- **Fallback:** SMS (SMSC.KZ)
- Автоматический fallback при ошибке WhatsApp

## Таблицы БД

**`users`** — основная таблица (phone = PK, role, name, surname, point_code, network_code, active)

**Redis-ключи:**
- `auth:staff:confirm:{phone}` — данные регистрации (имя, роль, точка, сеть). TTL: 24ч.
- `auth_token:{token}` — метаданные токена (phone, point, network, purpose). TTL: 24ч.

## Ключевые особенности

1. **Пользователь не создаётся в БД до подтверждения.** Всё живёт в Redis до шага 2.
2. **Promotion pattern.** Если клиент уже существует — он "промоутится" до staff, а не создаётся дубликат.
3. **Пароль устанавливает сотрудник**, не менеджер.
4. **Токен одноразовый** — удаляется после использования.
5. **Multi-tenant изоляция** — менеджеры могут приглашать только в свою точку, владельцы — в любую точку своей сети.
