# Gap Analysis: Missing Features in Refactor (vs Main Branch)

**Дата:** 19.02.2026
**Объект:** Сравнение `full-refactor` и `main`

---

## 1. Пропуски в Бизнес-логике (Major Gaps)

### 👥 Система приглашений (Staff/Management Invites)
На `main` реализован сложный асинхронный флоу, которого сейчас **нет** в рефакторе:
- **Link-based Invites:** Генерация уникальных 10-символьных токенов вместо 6-значных OTP.
- **Двухшаговая регистрация:** Менеджер создает инвайт -> Сотрудник сам ставит пароль по ссылке.
- **Promotion:** Логика превращения существующего `User` (клиента) в `Staff` при принятии приглашения.
- **Redis-состояние:** Хранение метаданных инвайта (`auth_token:{token}`) до момента подтверждения.

### 📋 Услуги и Мастера (Master Services)
В рефакторе есть только доменные структуры, на `main` это рабочая фича:
- **Связь Master-Service:** Индивидуальные цены и длительность услуги для каждого мастера.
- **Каталог:** Полный CRUD категорий и подкатегорий услуг.
- **Фильтрация:** Поиск мастеров по конкретной услуге и наоборот.

### 🏢 Управление Сетями (Networks)
- **Иерархия:** На `main` есть выделенный сервис `networks`, который управляет списком точек. 
- **Права доступа:** Логика `NetManager`, который видит данные всей сети, в рефакторе пока не реализована в `Policy`.

### 📅 Календарь и Записи (Booking)
- **Календарь:** На `main` есть хендлеры для отображения сетки записей.
- **Workflow записи:** Статусы записей, история изменений (Status History), которые на `main` уже работают через API.

## 2. Инфраструктурные пропуски (Technical Gaps)

### 🖼️ Управление Медиа (Media Context)
Согласно `project_docs/MEDIA_MIGRATION_README.md`, на `main` заложена (или реализована) сложная схема:
- **Junction Tables:** `user_media`, `staff_media`, `point_media`.
- **Логика Primary фото:** Автоматический выбор главного фото для точки.
- **Обработка статусов:** `pending`, `processing`, `active` для файлов в S3.
*В рефакторе сейчас реализован только базовый клиент S3.*

### 🔔 Продвинутые Уведомления
Согласно `project_docs/NOTIFICATION_IMPLEMENTATION.md`:
- **Recipient Types:** Разная логика и шаблоны для Клинта и Мастера.
- **Fallback Logic:** WhatsApp -> SMS не просто как код, а как полноценный сервис с очередью (Outbox).

### 🛠️ Формы и Кастомные данные
- На `main` есть модуль `forms`, позволяющий динамически собирать данные. В рефакторе этот контекст полностью отсутствует.

---

## 3. Различия в реализации (Architectural Divergence)

| Фича | Реализация в Main | План в Refactor |
|------|-------------------|-----------------|
| **Приглашения** | Ссылка (10 симв) | OTP (6 цифр) — *нужно совместить* |
| **Ошибки** | Простой маппинг | Строгий `ERROR_MAPPING_PATTERN` |
| **Роли** | Строки в БД | Value Objects + Policy Layer |
| **Тесты** | Мало | Обязательное покрытие UseCase |

---

## 4. Рекомендации по очередности переноса

1. **Staff Invite (UseCase):** Это критический функционал для запуска. Нужно адаптировать Link-based флоу под новую архитектуру.
2. **Master Services:** Без этого невозможно будет реализовать запись (Booking).
3. **Media Repository:** Перенести схему из `MEDIA_MIGRATION_README.md` в новые репозитории.
4. **Policy Expansion:** Расширить `Policy` для поддержки `NetManager` и `Manager` из `main`.
