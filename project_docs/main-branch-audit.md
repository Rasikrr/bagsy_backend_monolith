# Audit: Current Features on 'main' Branch

**Дата:** 19.02.2026
**Статус:** Анализ перед миграцией на новую архитектуру (DDD/UseCase)

---

## 1. Архитектурный стиль (Current)
Ветка `main` использует классическую трехслойную структуру (Layered Architecture):
- **Transport:** `internal/ports/http/handlers/`
- **Application:** `internal/services/` (Огромные сервисы, содержащие всю логику)
- **Domain:** `internal/domain/` (В основном простые структуры и перечисления)
- **Infrastructure:** `internal/repository/`

---

## 2. Реализованный функционал (Features)

### 🔐 Аутентификация и Управление паролями (Auth Context)
- **Login/Logout/Refresh:** По номеру телефона и паролю (bcrypt). JWT ротация.
- **Password Reset:** Двухшаговый сброс пароля (запрос ссылки -> подтверждение).
- **Verify Auth Token:** Специальный эндпоинт для проверки временных токенов (например, для регистрации приглашенных).

### 👥 Управление персоналом (Staff Management Flow)
Это ключевое отличие `main` от текущей ветки рефакторинга.
- **Staff Invite (Link-based):**
    - `RegisterStaff`: Менеджер создает инвайт для сотрудника.
    - `RegisterStaffConfirm`: Сотрудник подтверждает регистрацию, устанавливая свой пароль.
    - `RegisterStaffResend`: Повторная отправка инвайта.
- **Management Invite:**
    - `RegisterManagement`: Регистрация менеджеров в сеть/точку.
    - Аналогичные флоу подтверждения и переотправки.
- **Promotion Pattern:** Логика автоматического превращения обычного пользователя (клиента) в сотрудника при подтверждении инвайта.

### 🏢 Организационная структура
- **Networks (Сети):** На `main` уже есть зачатки работы с сетями (`internal/services/networks/`).
- **Points (Точки):** Логика создания и получения точек (локаций).
- **Bagsies (Организации):** Управление основным профилем организации.

### 📅 Календарь и Записи
- **Calendar:** На `main` присутствуют хендлеры для работы с календарем (`internal/ports/http/handlers/calendar/`).
- **Services:** Управление услугами и их категориями.
- **Master Services:** Привязка услуг к конкретным мастерам.

### 📋 Формы (Forms)
- Кастомные формы для сбора данных при записи или регистрации.

---

## 3. Технические детали (Infrastructure)
- **Redis:** Активно используется для хранения временных инвайт-токенов (`auth_token:{token}`) и данных регистрации (`auth:staff:confirm:{phone}`).
- **Notifications:** Интеграция с WhatsApp и SMS для отправки ссылок на регистрацию.
- **Domain:** Использование `enumer` для перечислений (например, `TokenPurpose`).

---

## 4. Что нужно учесть при рефакторинге (Gap Analysis)
1. **Invite Flow:** На `main` реализован сложный флоу инвайтов сотрудников через токены. В новой архитектуре (UseCase) мы пока сосредоточились на регистрации владельца через OTP. Нужно будет перенести логику Link-based инвайтов.
2. **Promotion:** Логика "пользователь -> сотрудник" уже отлажена на `main`, её важно не потерять.
3. **Networks:** На `main` уже есть разделение на сети и точки, в новой ветке пока только локации.
4. **Calendar:** На `main` уже есть API для календаря, которые нужно будет переписать на новые UseCase'ы.
